Server-side Support for Video Streaming

Posted on 2019-02-07 | Edited on 2019-02-16

In the last few days, I was busy with implementing functionalities of video uploading and online playing. The video streaming part impeded me for about 3 days. To fix it, I studied some HTTP headers which I didn’t use before. And to solve memory issue, I reviewed basic knowledges of Java IO. In general, it’s a good experience of learning and problem solving.

Download vs. Online Play

In our daily use of browser, we should have noticed that when the browser opens urls of files, it’ll either download the content or try to display it. Actually, this action is controlled by the response header Content-Disposition.

Response header: Content-Disposition

  • Content-Disposition: inline: inline tells the client side to display/open the file. And that’s what I should use for this online playing task.

  • Content-Disposition: attachment： This will trigger file downloading

  • Content-Disposition: attachment; filename="filename.jpg": We can set the name of the file. It’s also used in our project. When we upload the file, the file is saved with unique UUID as filename to avoid name conflicts, and at the same time, we persist original filename to database. As we retrieve the file, we use this header to set the filename that comes from database. In this process, we don’t lose the file’s extension.

Memory Issue

In my 1st round of implementing it, I copied the code from our existing ‘getImage’ functions since they both retrieve bytes from Minio (our file server). I just wanted it to work and might refactor the code later. My laziness brought me the memory issue.

When the server was handling multiple video playing requests, the system ran into the error, java.lang.OutOfMemoryError: Java heap space.

tip: we can use top command to check memory status

I went over the code of reading InpuStream and found that it was using IOUtils.toByteArray() the function that reads all bytes into an array one time. So, whenever the client requests for a video file, it gonna be loaded into the memory. It’s fine with small size images but costing too much memory for video files.

To improve this, I swtiched it to the classic way of reading InputStream, which uses a while-loop to read bytes into buffer and writes bytes into the OutputStream. We can write clean code with Java 7 try-with-resources statement instead of redundant finally + try-catch code block.

Not playable on IOS and Safari ->Add Range request header

For test convenience, I changed the backend filter to let video GET call pass through, then I could test in the browser from any device. The video was successfully played on Chrome. But when we tried to play videos on IOS device, it didn’t work. And I tested in Safari, it also didn’t work. Then, I was pretty sure it’s caused by some special requirement from Apple.

After doing some research, I found that IOS devices need video resource to support partial response. To play a video, IOS device will send a ‘testing’ request first. That request contains a Range header likeRange: bytes=0-1 , to see if the server support partial content and also get length of the file to send ranged request in the following steps.

Partial Content

The following part is about partial content required request and repoonse headers.

Range Header

more details

No Support：Accept-Ranges: none

Content-length Header

more details

total length of the file

Content-Range Header

more details

the range of returned data in the response

Status code 206: Partial Content

more details

Tell the client side the response contains partial content and the client side need to use Content-length and Content-Range to check if the whole file is received.

Range header lost after deployed

I was thinking that’s the end of this task. However, when the backend was deployed on Amazon server, the video playing stopped working again on IOS device.

Since the same video could be played on Chrome, I speculated that the Range header was not working. After checking the log files, I found that all Range headers are null after going through Nginx. Our Nignx gateway was not redirecting the Range header.

To fix it, I added this line to Nginx configuration:

proxy_set_header Range $http_range;

Learnt from it

  • Don’t directly reuse others’ functions without understanding what it is doing.
  • Create a convenient testing environment when tackling an issue. (Like changing filter rule to test on different platform easily)
  • Learn to use tools. (like using JProfiler to monit memory issue)
  • There’s still a lot I can learn from HTTP

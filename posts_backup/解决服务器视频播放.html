服务器视频在线播放

最近忙了几天一直在做服务器视频上传和在线播放视频的功能，然后一直卡在在线播放，花了不少时间学习HTTP还有基础的Java IO， 部署到服务器后，也遇到一些Nginx的问题。

过程中加深了很多HTTP的知识，HTTP真的是神协议，易懂高效，也许这就是它被大家接受的原因吧。

下载 VS 播放

经常我们会发现，点开一个链接可能是直接下载文件，或者是浏览器尝试打开这个文件。这个部分需要用到HTTP header：Content-Disposition

Content-Disposition

详细解释

Content-Disposition: inline: 代表的是可以使用网页展现（用浏览器打开），在这次任务中我就选用的inline，因为想让浏览器在线播放。

Content-Disposition: attachment： 浏览器会直接下载文件

Content-Disposition: attachment; filename="filename.jpg":指定文件名，这次项目也用到了，因为可能存文件的时候确保唯一性，文件名是唯一的ID，原文件名存在了数据库。下载文件的时候，可以服务器从数据库把名字找回来，带着文件后缀很方便，直接交给浏览器就行了。

内存问题

第一次做这个部分的时候，我直接从下载图片功能复制了代码，是以前别的同事写的。跑是可以跑得起来，但是服务器处理多个请求的时候内存爆增，读大的视频文件后来直接内存满了，速度非常慢。我回头读代码，发现是把整个文件一下子读到一个byte[]的方法，这种方法对小的图片可能无所谓，但大的文件会一下子把内存占满。后来我就改成了经典的分次读取，顺便用了Java7的try语言特性，不需要像以前一样用finally和try-catch额外处理InputStream和OutputStream的关闭。

Safari 和 IOS 播放不了 -> 使用Range头

为了方便测试，我把后端filter规则改了一下，然后所有视频请求都可以通过，可以直接在浏览器打开。成功在Chrome上播放起来后，心里也产生了一个疑问，请求的视频文件都是一整个传给客户端的。按道理应该是会分段的呀，想着想着用我们的App端测试了一下，问题来了，都是没法播放，IOS的浏览器也无法打开，电脑端的Safari浏览器也无法播放。查看了一下发送的请求，发现苹果端都会先发一个请求包含Range头，并且第一个Range头永远是Range: bytes=0-1

后来经过研究，苹果设备的视频播放默认需要服务器支持Range头，第一次会发送一个0-1 byte的请求，服务器需要表明支持分段传输，并且Response的status code是206，所以我需要改进代码，让他支持标准的Range头和分段传输。

分段传输

Request

Range Header

详细解释

说明请求的数据单位和范围

Response

Accept-Ranges Header

详细解释

表明服务器是否支持分段

支持的话：Accept-Ranges: bytes

不支持：Accept-Ranges: none

Content-length Header

详细解释

要传输文件的总长度

Content-Range Header

详细解释

返回部分在目标文件中的范围

Status code 206: Partial Content

详细解释

表明返回数据是分段数据，client端需要通过Content-length和当前的Content-Range判断是否需要继续请求。

坑：部署后丢失Range头

在本地测试后一切OK，部署到Amazon后，发现IOS视频又播放不了了，同样的链接可以在Chrome打开。初步判断是Range头出问题了，从服务器的LOG看到，Range头全部都是null, 因为本地从来没有出现过这种情况，所以猜测是Nginx没有转发Range头。

参考这篇博客后，在Nginx的配置文件对应的location里面添加:

proxy_set_header Range $http_range;

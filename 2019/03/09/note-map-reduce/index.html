MapReduce Note

Posted on 2019-03-09 | In Note

Notes for the MapReduce Paper

List bullet points for quick reviews in future.

Prgoramming Model

  • Map takes an input pair and produces a set of intermediate key/value pairs.

    map (k1, v1) -> list (k2, v2)

  • The Reduce function accepts an intermediate key I and a set of values for that key. Typically zero or one output value.

    reduce (k2, list(v2)) -> list (v2)

Implementation

Execution Overview

Overall flow of a MapReduce opeartion:

  1. User MapReduce program: Splits the input files into M pieces of typically 16 megabytes to 64 megabytes per piece.
  2. One of the copies of the program is special - the maser. There are M map tasks and R reduce tasks to assign.
  3. Map worker parses key/value pairs out of the input data and passes each pair to the user-defined Map function. The intermediate key/value pairs produced by the Map function are buffered in memory.
  4. Map worker: Buffered pairs are written to local disk, partitioned into R regions by the partitioning function. The locations of these buffered pairs are passed back to the master.
  5. Reduce worker: When a reduce woker is notified by the master about these locations:
    1. Uses RPC to read the buffered data from the local disks of the map workers.
    2. Sorts all intermediate data by keys to group all occurrences of the same key together.
  6. The Reduce worker iterates over the sorted intermediate data and for each key, it passes the key and corresponding set of intermediate values to the user’s Reduce function.
  7. When all map tasks and reduce tasks have been completed, the master wakes up the user program.

Master Data Structures

  • For each map task and reduce task, it stores the state (idle, in-progress, or completed), and the identity of the worker machine (for non-idel tasks).
  • For each completed map task, the master stores the locations and sizes of the R intermediate file regions produced by the map task.

Fault Tolearance

Worker Failure

The master pings every worker periodically.

  • no response -> mark as failed
  • any map task completed by the worker are reset back to their initial idle state.
  • any map task or reduce task in progress on a failed worker is also reset to idle and become eligible for rescheduling.
